01 — SYSTEM OVERVIEW

What AppDynamics Actually Watches

APM stands for Application Performance Monitoring (some say Management) — the discipline of watching how software actually behaves in production, and catching the failure before your users report it for you. AppDynamics is a suit built for exactly that: instrumented sensors on every tier, feeding one brain, so you can find the bottleneck before it finds you.

Application

Code Layer

Profiling tools instrument the running code itself — methods, threads, exceptions — so you see exactly what the application is doing, not just that it's slow.

Infrastructure

Infra Layer

Tools like perfmon, nmon, vmstat, and SiteScope watch the hardware and OS underneath — CPU, memory, disk I/O, network — the plumbing the app depends on.

Database

Data Layer

DB profilers, AWR reports, and ADDM reports find the query that's actually holding everything up — locks, waits, and execution plans included.

Browser

End-User Layer

Dev tools, WebPageTest, and GTmetrix show what the person on the other end of the connection actually experiences while the page loads.

WHY IT ALL MATTERS: MTTR

Collecting metrics is easy. Finding the root cause fast is the job. AppDynamics and tools like it exist to shrink MTTR — Mean Time To Repair (or Recovery) — the clock that starts the second something breaks and doesn't stop until it's fixed. A tool stack that gives you dashboards but no root cause is a suit with lights and no repulsors.

RUM

Real User Monitoring

Passive observation of actual live traffic — real people, real browsers, real network conditions — hitting your application right now.

Synthetic

Synthetic Monitoring

A scripted job that launches a scripted browser flow on a schedule, simulating a user journey automatically — so you catch a problem even at 3 a.m. when no real traffic is flowing.

02 — ANATOMY OF THE SUIT

Controller, Agents & the Nervous System

Two deployment flavors exist: SaaS, where AppDynamics hosts the Controller for you and you just launch its URL, and On-Premise, where you run the whole stack yourself via the Enterprise Console. Either way, the anatomy is the same.

The Brain

Controller

The central nervous system. Stores configuration, tiers, nodes, health rules, and metric history, and serves the dashboard UI you actually click around in. Nothing works without it.

The Memory

Events Service

A dedicated store built for high-volume events — health rule violations, transaction snapshots, logs — so the Controller doesn't drown trying to hold everything itself.

The Nerve Endings

Agents

Small processes embedded in your tiers — Java, Machine, DB, or Browser — that instrument what's actually happening and phone it home to the Controller in real time.

On-Prem Only

Enterprise Console

The installer-of-installers for self-hosted deployments — it's what stands up and manages the Controller, MySQL, Events Service, and Search Service on your own infrastructure.

03 — SUIT ASSEMBLY

Installing the Controller on Ubuntu 22.04

This is the on-premise path — bolting the reactor into the chest plate yourself. Ten steps, start to finish, from a bare Ubuntu box to a running Controller with your first application defined.

Prepare Ubuntu 22.04

Update the box and make sure the environment is ready before anything else gets installed.

sudo apt update && sudo apt upgrade -y
  • Reboot after the upgrade
  • Make sure your hostname is set — the Controller uses it internally
  • Install a Java runtime if your environment requires one

Create a Controller User Recommended

The Controller should run under its own dedicated system user, not root.

sudo useradd -m -d /home/appd -s /bin/bash appd sudo passwd appd sudo usermod -aG sudo appd

Download the Controller Installer

Grab the Linux on-premise Controller .bin installer from the AppDynamics download portal.

cd /opt/appdynamics
  • Log in to the AppDynamics download portal
  • Download controller_64bit_linux-x.x.x.x.bin
  • Move it into /opt/appdynamics/

Make the Installer Executable

It's a self-extracting binary — it needs permission to run.

chmod +x controller_64bit_linux-x.x.x.x.bin

Run the Controller Installer Critical

This single step installs the Controller, MySQL, the Events Service, and the Search Service together.

sudo ./controller_64bit_linux-x.x.x.x.bin
  • Accept the license agreement
  • Choose the default ports unless you have a reason not to
  • Provide the appd user you created earlier
  • The installer configures MySQL automatically

Start Controller Services

Bring everything online and confirm it actually stayed online.

sudo service appdcontroller start
  • Verify MySQL is running
  • Verify the Events Service is running
  • Check logs in /opt/appdynamics/controller/logs/

Access the Controller GUI

Open the dashboard in a browser for the first time.

http://<your-server-ip>:8090/controller
  • Log in with the default admin account created during install
  • Verify the Controller loads cleanly

Apply Your License Required

The Controller won't accept agent connections without one.

  • Upload the license file in the Controller GUI
  • Verify which modules it unlocks — APM, Machine Agent, and so on

Create Your First Application

An application has to exist before any agent has somewhere to report to.

  • Go to Applications → Create Application
  • Name it — for example, UbuntuTestApp
  • Create a Tier and a Node underneath it

Install Agents on Linux

With the Controller live, the nerve endings can finally connect.

  • Download a Java Agent or Machine Agent
  • Configure controller-info.xml to point at your Controller
  • Restart the application so the agent loads
  • Confirm metrics start flowing into the Controller
04 — THE NERVE ENDINGS

Every Agent, and What It Actually Senses

A Controller with no agents is a helmet with no visor. Each agent type wires up a different sense — code execution, hardware, database internals, or the browser itself.

app tier

Java Agent

Attaches directly to the JVM of your application and instruments methods, threads, and calls in real time. This is what generates business transactions, call graphs, and JVM/heap data.

host / OS

Machine Agent

Watches the server itself — CPU, memory, disk I/O, network. Without it running, the Controller's "Servers" tab for that node simply stays empty, even if the app agent is reporting fine.

data layer

DB Agent

Runs as its own small process — java -jar db-agent.jar — and monitors database hardware and health: CPU, memory, I/O, live sessions, wait states, and connected clients.

front end

Browser / EUM Agent

Captures real end-user page-load experience (RUM) or drives a scripted flow on a schedule (Synthetic) — the only agent that sees the app the way an actual person does.

WORKED EXAMPLE: NODE.JS + EXPRESS

Set your application name and tier name (say, express), install the app agent, then paste your account's license line on the first line of your entry file. Start the process with node app.js — the moment it boots, the tier lights up in the Controller. The same instinct applies whether it's Java, Node, or a DB agent: configure it, point it at the Controller, restart the process, and watch for the first metric to land.

05 — FLYING THE SUIT

Navigating the Portal Like a Pro

This is the HUD. Once agents are reporting, everything below is where you actually live day to day — from the first glance at the flow map to a full root-cause war room.

5.1

Dashboard & Application Flow Map

The flow map is the suit's radar: a live diagram of every tier, every node, and the traffic moving between them right now. It's usually the first thing you open, because it's the fastest way to spot where activity is concentrated — or where it's suspiciously silent.

5.2

Tiers & Nodes

A Tier is the high-level entity — a logical service, like "checkout-api." A Node is a specific running instance configured underneath that tier — one JVM, one process. One tier can have many nodes; that's how you scale horizontally and still see it all as one thing.

5.3

Business Transactions

A business transaction is a real user-facing call — a web request or REST API hit — tracked end to end across every tier it touches. The scorecard view color-codes health at a glance:

Healthy — within baseline
Slow — response time drifting
Errors — failing or breaching threshold

Load a few thousand threads through a tool like JMeter and watch the errors-per-minute column and response time column move together — that relationship is the whole story of "is this holding up under traffic."

5.4

Transaction Snapshots & Call Graphs

A snapshot is a recorded execution trace of one specific transaction instance, captured over a monitoring window so you can drill into exactly what happened, method by method — line number, class, type, and total time spent.

Full call graph — complete instrumented trace
Partial call graph — still enough to point at the issue

Even a partial call graph is useful: AppDynamics summarizes it on the transaction overview page and flags the likely problem area, so you're not left with nothing just because the full trace wasn't captured — though drill-down only works for tiers that actually have an agent configured, not the ones sitting dark.

5.5

Health Rules & Violations

A health rule defines a threshold against a baseline — for example, average response time within a normal band versus a critical band that might be 3 or 5 seconds depending on the transaction. Breach it, and a violation fires at the transaction level; you can wire automated actions to run the moment that happens.

Baseline / HealthyWarningCritical

Anomaly detection layers machine learning on top of static thresholds, catching drift that a fixed number would miss entirely.

5.6

JVM, Heap & Garbage Collection

For Java tiers, the JVM's memory story splits into two halves — and knowing which half is leaking tells you what kind of problem you actually have.

Heap Memory

  • Young Generation — short-lived objects
  • Old Generation — long-lived survivors
  • Garbage collection reclaims unreferenced objects here

Non-Heap Memory

  • Permanent Generation / Metaspace
  • Class metadata and JVM internals live here
  • A leak here looks very different from a heap leak

Major and minor garbage collection cycles run at different frequencies and cost — automatic leak detection watches the pattern of GC activity over time, and a heap dump lets you freeze the moment and inspect exactly what's being held in memory and why.

5.7

Diagnostic Sessions & War Rooms

When a health rule breaches and the standard snapshot isn't enough, start a diagnostic session to capture deeper data — including thread and heap dump analysis — for the specific window when things went wrong.

A War Room is the collaborative space for what happens next: pull in the right people, attach the relevant charts and documents, and work the root cause analysis (RCA) together in one place instead of six different Slack threads.
